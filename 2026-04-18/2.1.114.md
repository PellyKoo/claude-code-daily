## 1. Claude Code 创造者的"极简"工作流
- **来源**: [Boris Cherny](https://x.com/bcherny/status/2007179832300581177) on X | 🔥 高互动热帖
- **摘要**: Claude Code 的创造者 Boris Cherny 公开分享了他的日常使用方式，令人意外的是非常"原味"——几乎不做自定义配置。核心技巧：通过 tmux 连接到 Mac Mini 远程开发，WiFi 断开后重连会话完好无损；使用 `/permissions` 命令预先放行信任命令以保持心流状态。证明了 Claude Code 开箱即用的强大能力。
- **代码示例**:
```bash
# 远程开发工作流：tmux + Mac Mini
# 在 Mac Mini 上启动 tmux 会话
tmux new -s claude-dev

# 从笔记本 SSH 连接
ssh mac-mini -t "tmux attach -t claude-dev"

# WiFi 断开？重连即可，会话完整保留
# 使用 /permissions 放行常用命令
/permissions
# 添加: npm test, npm run build, git status 等
```

## 2. Matt Van Horn 的 Claude Code 全技巧合集
- **来源**: [Matt Van Horn](https://x.com/mvanhorn/status/2035857346602340637) on X | 🔥 大量收藏与转发
- **摘要**: 一篇全面的 Claude Code 技巧总结帖，涵盖从基础到进阶的所有实用 hack。关键亮点包括：善用 git worktree 开启 3-5 个并行 Claude 会话处理不同任务；在一次性环境中使用 `--dangerously-skip-permissions` 加速迭代；用 `/compact` 手动触发上下文压缩避免会话退化。被社区称为"Claude Code 速查手册"。
- **代码示例**:
```bash
# 并行工作流：git worktree + 多 Claude 会话
git worktree add ../feature-auth feature/auth
git worktree add ../feature-api feature/api
git worktree add ../bugfix-perf bugfix/perf

# 每个 worktree 启动独立 Claude 会话
# Terminal 1:
cd ../feature-auth && claude
# Terminal 2:
cd ../feature-api && claude
# Terminal 3:
cd ../bugfix-perf && claude

# 完成后清理 worktree
git worktree remove ../feature-auth
```

- **来源**: [Santiago](https://x.com/svpino/status/2029645889980305777) on X | 🔥 高互动技术演讲

- **代码示例**:
```

# 先规划再编码
> 阅读 @CLAUDE.md 和 @src/api/，
  然后制定重构 REST 端点的详细计划，
  先不要写代码

# 避免幻觉：明确指定文件路径
> 修复 @tests/auth.test.ts 中失败的测试用例
# 而不是说 "修复认证测试" (Claude 可能找错文件)
```

## 4. 50 条 Claude Code 每日最佳实践
- **来源**: [Vishwas (Codevolution)](https://x.com/CodevolutionWeb/status/2034683638382506063) on X | 🔥 高互动教程帖
- **摘要**: 系统整理的 50 条 Claude Code 日常使用最佳实践，从入门到进阶全覆盖。精华建议包括：短会话优于长马拉松（上下文退化是最大敌人）；在自主执行前先 commit 检查点，结果不对时回滚比"修复前进"成功率高得多；善用 `/memory` 命令将关键上下文持久化到跨会话记忆文件。
- **代码示例**:
```bash
# 检查点工作流：先 commit 再让 Claude 自主操作
git add -A && git commit -m "checkpoint: before claude refactor"

# 让 Claude 执行复杂重构
> 重构整个认证模块，迁移到 OAuth2

# 如果结果不理想，直接回滚重来
git reset --hard HEAD~1
# 重新开始一个新会话（比在退化会话中修复更高效）

# 使用 /memory 持久化关键信息
/memory
# 保存: "项目使用 PostgreSQL 15, 部署在 AWS ECS"
```

## 6. "穴居人模式"——节省 75% 以上用量
- **来源**: [Nathan Onn](https://www.nathanonn.com/claude-code-caveman-mode/) on Blog | 🔥 Reddit 热议话题
- **摘要**: 一种极端但有效的成本优化策略：在 CLAUDE.md 中指示 Claude 以最简洁的方式回复——不解释、不道歉、不冗余输出，只给代码和必要信息。配合 `/compact` 频繁压缩上下文，实测可节省超过 75% 的 token 用量。适合个人开发者或有严格预算限制的团队。
- **代码示例**:
```markdown
# CLAUDE.md 中添加穴居人模式指令
## Communication Style
- NEVER explain what you're about to do
- NEVER apologize or use filler phrases
- NEVER repeat back the requirements
- Code only, no prose
- If asked a question, answer in ≤10 words
- Use /compact after every 3-4 exchanges
```

## 7. 多代理 Opus+Sonnet 分层模式
- **来源**: [r/ClaudeAI](https://www.reddit.com/r/ClaudeAI/) on Reddit | 🔥 Reddit 高赞讨论
- **摘要**: 社区最受推崇的进阶工作流：主会话使用 Opus 处理复杂推理和架构决策，子代理使用 Sonnet 处理聚焦型任务（如代码搜索、测试编写），通过 `CLAUDE_CODE_SUBAGENT_MODEL` 环境变量控制。这样既保持了关键决策的质量，又大幅降低了整体成本。Reddit 共识：这是"性价比最高的 Claude Code 配置"。
- **代码示例**:
```bash
# 设置子代理使用 Sonnet（主会话默认 Opus）
export CLAUDE_CODE_SUBAGENT_MODEL="claude-sonnet-4-6"

# 启动 Claude Code（主会话使用 Opus）
claude

# Claude 在执行以下操作时会自动使用 Sonnet 子代理:
# - 代码搜索和探索 (Agent tool with subagent)
# - 并行研究任务
# - 简单的代码生成
# 而保留 Opus 用于:
# - 架构决策
# - 复杂 bug 诊断
# - 代码审查和重构方案制定
```

## 8. 上下文工程取代提示工程——2026 年新共识
- **来源**: [r/ClaudeAI](https://www.reddit.com/r/ClaudeAI/) + [多位开发者](https://x.com/milesdeutscher/status/2014027971254952332) on X | 🔥 跨平台热议
- **摘要**: 2026 年社区最大的范式转变：模型能力已不再是瓶颈，上下文管理才是真正的杠杆。实践要点：CLAUDE.md 控制在 200 行以内（超过则 Claude 开始忽略指令）；使用分层目录结构管理规则；从简单规则开始、迭代添加；用 hooks 执行确定性操作；包含构建命令、架构说明和编码约定。简言之：喂给 Claude 什么比怎么说更重要。
- **代码示例**:
```markdown
# 高效 CLAUDE.md 模板 (控制在 200 行以内)

## Build & Test
- `npm run build` - 构建项目
- `npm test` - 运行测试
- `npm run lint` - 代码检查

## Architecture
- src/api/ - REST 端点 (Express)
- src/models/ - 数据模型 (Prisma)
- src/services/ - 业务逻辑

## Conventions
- TypeScript strict mode
- 函数式组件 + hooks
- 错误统一用 AppError 类

## Do NOT
- 不要修改 migrations/ 下的已有迁移文件
- 不要在 service 层直接访问 req/res 对象
```
