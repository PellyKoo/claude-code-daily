## 1. 上下文管理黄金法则：70% 警戒线
- **来源**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) on GitHub | ⭐ 3.9k stars
- **摘要**: 当上下文使用率达到 70% 时，Claude 的精准度开始下降；达到 85% 时，幻觉明显增加。社区推荐在 70% 时使用 `/compact` 压缩上下文，90% 时强制执行 `/clear` 重新开始。经验丰富的用户甚至将目标控制在 30% 以下。
- **代码示例**:
```bash
# 上下文管理策略
# 70% 时 → 压缩上下文，保留关键信息
/compact

# 90% 时 → 清理并用简报重新开始
/clear
> 我正在实现用户认证模块，已完成 JWT 签发逻辑，
  接下来需要实现 token 刷新和权限中间件。

# 日常习惯：在高风险切换前用 /clear + 简报
```

## 2. 子代理隔离策略：用 Subagent 管理上下文污染
- **来源**: [shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice) on GitHub | ⭐ 47.8k stars
- **摘要**: 子代理的上下文是隔离的，只有最终结论会返回主会话。利用这一特性，把探索性任务（搜索代码、调查 bug）派给子代理，避免大量中间结果污染主上下文。一个代理负责构建，另一个在全新上下文中找 bug，实现"测试时计算"分离。
- **代码示例**:
```markdown
# 在 Claude Code 中使用子代理模式

> 请用子代理搜索项目中所有使用 deprecated API 的文件，
  只返回文件列表和修改建议摘要。

# 或者使用 fork 上下文的 skill：
# context: fork 让 skill 在隔离子代理中运行
# 主上下文只看到最终结果

# 实战：双代理 review 模式
> 代理 A：实现这个功能
> 代理 B（新上下文）：审查代理 A 的实现，找出潜在问题
```

## 3. CLAUDE.md 分层管理：rules 目录拆分大型配置
- **来源**: [shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice) on GitHub | ⭐ 47.8k stars
- **摘要**: CLAUDE.md 应控制在 200 行以内。当规则过多时，使用 `.claude/rules/*.md` 目录拆分指令，每个文件专注一个领域。配合 `<important if="...">` 标签实现条件触发，防止指令被忽略。
- **代码示例**:
```
项目结构：
├── CLAUDE.md                    # 核心规则（< 200 行）
├── .claude/
│   └── rules/
│       ├── code-style.md        # 代码风格规范
│       ├── testing.md           # 测试策略
│       ├── security.md          # 安全检查清单
│       └── api-conventions.md   # API 设计约定

# CLAUDE.md 中使用条件标签：
<important if="editing files in src/auth/">
所有认证相关修改必须添加单元测试，覆盖率不低于 80%
</important>

<important if="creating new API endpoints">
必须遵循 RESTful 规范，包含 OpenAPI 文档注释
</important>
```

## 4. /rewind 替代纠正：撤销失败尝试重新出发
- **来源**: [shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice) on GitHub | ⭐ 47.8k stars
- **摘要**: 当 Claude 的实现方向错误时，不要在错误基础上纠正，而是用 `/rewind` 撤销失败的尝试。关键技巧：在 rewind 之前先让 Claude "从这里总结"，生成交接笔记，这样新的迭代能带着经验教训重新开始，避免重蹈覆辙。
- **代码示例**:
```bash
# 错误做法：在失败基础上不断修补
> 这个方案不对，改用 Strategy 模式重写  # 上下文被污染

# 正确做法：总结经验 → 撤销 → 重新开始
> 请总结当前实现的问题和学到的经验
# Claude 输出总结...

/rewind  # 撤销到失败之前的状态

> 上次尝试的经验：[粘贴总结]
  请用 Strategy 模式重新实现，避免之前的问题
```

## 5. MCP 服务器安全审查：5 分钟审计清单
- **来源**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) on GitHub | ⭐ 3.9k stars
- **摘要**: 社区已发现 24 个 CVE 和 655 个恶意 skill。安装任何 MCP 服务器前，执行 5 分钟审计：验证 GitHub 来源可信度、审查代码确保最小权限、白名单限制文件系统访问、先在 Docker 隔离环境中测试。
- **代码示例**:
```bash
# MCP 服务器安全审计流程

# 1. 验证来源
gh repo view <mcp-server-repo> --json stargazerCount,updatedAt,license

# 2. 检查权限范围
grep -r "filesystem\|exec\|spawn\|child_process" node_modules/<mcp-server>/

# 3. Docker 隔离测试
docker run --rm -v $(pwd):/workspace:ro \
  --network none \
  <mcp-server-image> --test

# 4. 白名单配置（.claude/settings.json）
{
  "mcpServers": {
    "my-server": {
      "command": "node",
      "args": ["server.js"],
      "allowedPaths": ["/workspace/src/**"]
    }
  }
}
```

## 6. 权限通配符配置：减少 84% 的权限弹窗
- **来源**: [shanraisshan/claude-code-best-practice](https://github.com/shanraisshan/claude-code-best-practice) on GitHub | ⭐ 47.8k stars
- **摘要**: 使用通配符语法（`Bash(npm run *)`、`Edit(/docs/**)`）而非 `dangerously-skip-permissions`。配合 `/sandbox` 实现文件和网络隔离，权限弹窗减少约 84%。也可使用内置的 `/less-permission-prompts` skill 扫描历史记录自动生成推荐白名单。
- **代码示例**:
```json
// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(npx jest *)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Edit(src/**)",
      "Edit(tests/**)",
      "Read(**)"
    ]
  }
}
```
```bash
# 或使用内置 skill 自动生成
/less-permission-prompts

# 使用沙箱模式进一步隔离
/sandbox
```

## 7. 双模式工作流：初始化代理 + 编码代理分离
- **来源**: [HackerRank](https://x.com/hackerrank/status/2016398850265935929) on X | 🔥 高互动（官方技术账号分享）
- **摘要**: 将长时间运行的任务拆分为两个阶段：初始化代理负责项目搭建、功能列表、Git 仓库、进度追踪；编码代理负责读取、构建、测试、提交和更新。避免单一上下文承载过多状态，减少 token 消耗和上下文丢失。
- **代码示例**:
```bash
# 阶段一：初始化代理（设置基础架构）
claude "作为初始化代理：
1. 分析项目需求文档 requirements.md
2. 创建项目目录结构
3. 初始化 Git 仓库和 .gitignore
4. 生成功能清单到 TODO.md
5. 配置 CLAUDE.md 和基础测试框架"

# 阶段二：编码代理（逐个实现功能）
claude "读取 TODO.md，逐个实现功能。
每完成一个功能：运行测试 → 提交代码 → 更新 TODO.md"
```

## 8. 渐进式采纳策略：从简单开始，两周后再加复杂配置
- **来源**: [FlorianBruniaux/claude-code-ultimate-guide](https://github.com/FlorianBruniaux/claude-code-ultimate-guide) on GitHub | ⭐ 3.9k stars
- **摘要**: 社区共识：不要一开始就搭建复杂的代理系统。推荐路径：基础 CLAUDE.md + 几个常用命令 → 在生产环境测试两周 → 根据实际需要逐步添加 agents、skills 和 MCP 服务器。只有 30% 的用户显式设置了协作规则，这导致了大量的上下文漂移问题。
- **代码示例**:
```markdown
# 渐进式采纳路线图

## 第1周：基础配置
- 创建 CLAUDE.md（项目概述 + 代码风格）
- 熟悉 /compact、/clear、/rewind

## 第2-3周：命令与工作流
- 添加常用 slash commands
- 配置权限白名单
- 建立提交和测试习惯

## 第4周+：按需扩展
- 评估是否需要自定义 skills
- 考虑子代理并行开发
- 引入 MCP 服务器（先审计！）
```
